INTRODUCTION 1.1 Purpose This document fully describes the software requirements for the new VULE ITS Website Portal system. The SRS serves stakeholders including management, technical teams, UI/UX, QA/QC, and other relevant parties. Its purpose is to define clear and unified requirements, minimize misunderstandings, serve as a basis for design, development, and testing, and reduce requirement changes during implementation. 1.2 Scope The system consists of two main components: Public Website (Portal): Company introduction, products, news, contact, banners/slideshow, SEO, Privacy Policy, Terms of Service, About Page. Admin CMS: Product management, news management, banner/slider management, media management, account &amp; role management, rich text content entry, code embedding, About Page section management, Privacy Policy &amp; ToS management. 1.3 Out of Scope No e-commerce/cart functionality, online payment, loyalty, public account registration, CRM, content approval workflow, or multilingual support (to be considered in future phases if needed). 1.4 Definitions, Acronyms, Abbreviations Term Definition SRS Software Requirements Specification UI User Interface UX User Experience SEO Search Engine Optimization FE Frontend BE Backend CMS Content Management System CRUD Create, Read, Update, Delete RBAC Role-Based Access Control 1.5 References IEEE 830:1998 - Software Requirements Specification Next.js, Node.js, Tailwind CSS Documentation 1.6 Overview This document includes: Introduction, Overall Description, Specific Requirements (FR, NFR, Use Case), and Appendices. 2. OVERALL DESCRIPTION 2.1 Product Perspective The new Website Portal will completely replace the legacy system, following a decoupled FE-BE architecture. The frontend (FE) handles UI, SEO, and content rendering; the backend (BE) provides APIs, data management, and business logic. The CMS may be a module of the FE or a standalone system. 2.2 Product Functions Public: Homepage (banner, featured products, latest news) About page (rich text content, images, banners, SEO, dynamic sections) Product pages (list + detail, filtering, demo/consultation CTA) News pages (list + detail, pagination, tag filter) Contact page (form, DB storage, email, Google Maps) Advanced search (filter by content type, keyword highlighting) Privacy Policy page Terms of Service page Admin CMS: Product management (CRUD, code embedding, Active/Expired status) News management (CRUD, code embedding, tags, display time) Banner/Slider management (CRUD sliders, upload images, drag-drop, position, timing, accessibility) Media management (upload, compression, resize, WebP, folders) Account &amp; RBAC management (SuperAdmin, Admin, Editor) About Page section management (CRUD sections, drag-drop, rich text, SEO, media) Privacy Policy &amp; Terms of Service management (versioning, SEO meta, publishing) 2.3 User Classes and Characteristics User Type Description Needs Public User Customers, partners Attractive UI, easy information access, good SEO Admin Internal staff Powerful content management Editor Marketing, Sales Easy content creation/editing, code embedding Root Admin IT Full system control 2.4 Operating Environment FE: Chrome, Safari, Firefox, Edge BE: Node.js Database: MySQL HTTPS required, hosting must support SSR/SSG 2.5 Design Constraints Technologies: Node.js, Next.js, Tailwind CSS Decoupled FE-BE architecture UI must follow IT industry standards Content must allow HTML, CSS, JS embedding 2.6 Assumptions and Dependencies Company email server operates reliably Hosting supports SSR/SSG Content update frequency depends on Marketing 2.7 Business Rules Only content with Active status is displayed publicly Slugs must be unique File uploads &lt; 10MB News has start/end date; expired news is hidden About Page is single-instance SEO data is mandatory for all public pages Expired products are retained in CMS, not deleted Each user has only one role at a time Root Admin cannot be deleted 2.8 Data Constraints Max upload: 10MB/file Allowed mime types: jpg, png, webp, mp4, pdf, docx Title ≤ 150 characters, Description ≤ 300 characters Slugs must not contain special characters Publish date cannot be in the future 3.
